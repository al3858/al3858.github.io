Lab 2 — EC2, S3 & IAM Roles

PART 1

EC2 Web Server

Deployed an EC2 instance with Pulumi, created an HTML website hosted on port 8080, and verified public access via DNS.

PART 2

S3 + IAM Role

Attached an IAM instance profile so EC2 could read S3 content via IMDS temporary credentials — no hardcoded keys.

PART 3 (EXTRA CREDIT)

user_data Automation

Automated the entire server setup using EC2 user_data bootstrap script — eliminating manual SSH and configuration steps.

Key insight: By default, EC2 instances have no credentials and cannot access any AWS services (default-deny). The IAM role pattern fixes this — each instance only gets access to the specific bucket it needs, so even if something gets compromised, the damage stays limited. The IMDS temporary credentials (AccessKeyId starting with "ASIA") expire after a few hours and are never stored in code.

Lab 2 Key Findings

D8

IAM Role + IMDS Credential Chain

When the EC2 instance starts, it has an instance profile attached with an IAM role. The instance contacts the Instance Metadata Service (IMDS) at 169.254.169.254 to get temporary credentials. The boto3 library automatically uses these credentials — no access keys stored in code. The X-Served-From: S3 header proved success because it is only added after a successful S3 response.

D9

Default-Deny Security Model

When aws s3 ls returned "Access Denied" and aws configure list showed all values as None — that is default-deny. Nothing is allowed unless explicitly granted. If any EC2 instance could read any S3 bucket automatically, one hacked instance could access the entire data lake. IAM roles fix this by scoping access to only what each instance needs.

Lab 3 — VPC, Subnets & Network Architecture

PART 1

VPC + Bastion Host

Built a custom VPC with public/private subnets, Internet Gateway, NAT Gateway, route tables, security groups, bastion host, and private EC2 instance — all via Pulumi (21 resources in 2 min).

PART 2

SSH Jump + NAT Verification

SSH'd into bastion host, then hopped into the private instance. Verified private instance has no public IP but routes outbound traffic through the NAT Gateway (IP matched natEip).

Key insight: The private instance's outbound IP matched the NAT Gateway EIP (3.218.133.123). This happened because private instances have no public IP — all outbound traffic routes through the NAT Gateway. The bastion host pattern exists so we can access private servers without giving them public internet exposure.

Lab 3 Key Findings

D3

Internet Gateway vs NAT Gateway

An Internet Gateway is the VPC's connection to the internet used by public subnets — instances get public IPs and can send and receive traffic. A NAT Gateway is a one-way door for private subnets — instances can send outbound requests, but the internet cannot initiate connections back. Private instances cannot use the IGW directly because they have no public IPs.

D4

Bastion Host Pattern

The bastion host exists because private instances cannot be accessed directly from the internet. The bastion acts as an SSH entry point inside the public subnet — SSH into the bastion, then SSH into private instances. One improvement: restrict SSH access to specific IP addresses, or use AWS Systems Manager Session Manager instead of SSH to eliminate port 22 exposure entirely.
